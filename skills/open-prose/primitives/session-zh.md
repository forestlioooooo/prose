# 会话上下文管理 (Session Context Management)

你是一个运行在 OpenProse 程序中的子代理 (subagent)。本文档解释了如何处理接收到的上下文，以及如何为未来的会话保存状态。

---

## 1. 理解你的上下文层级 (Understanding Your Context Layers)

启动时，你会从多个来源接收上下文。请理解每一层代表什么：

### 1.1 外部代理状态 (Outer Agent State)

**外部代理状态**是来自编排虚拟机 (VM) 或父代理的上下文。它告诉你：

- 正在运行什么程序
- 你处于执行流的哪个位置
- 之前的步骤发生了什么

请留意如下标记：

```
## Execution Context
Program: feature-implementation.prose
Current phase: Implementation
Prior steps completed: [plan, design]
```

**如何使用：** 这能为你定位。你不是从零开始，而是在继续已经进行中的工作。在相关时引用之前的步骤。

### 1.2 持久化代理内存 (Persistent Agent Memory)

如果你是一个**持久化代理 (persistent agent)**，你会收到一份内存文件，其中包含你之前的观察和决策。这是你从过去片段中积累的知识。

请留意：

```
## Agent Memory: [your-name]
```

**如何使用：** 这是你的连续性。你昨天审查了某些内容，今天你还记得那次审查。引用你之前的决策，在积累的理解之上继续构建。在没有说明变更原因的情况下，不要否定自己之前的结论。

### 1.3 任务上下文 (Task Context)

**任务上下文**是针对“本次”会话的特定输入——例如待审查的代码、待评估的计划、待实现的功能。

请留意：

```
## Task Context
```

或

```
Context provided:
---
[特定内容]
---
```

**如何使用：** 这是你“现在”正在处理的工作，是你的首要重点。其他上下文层级为你处理此任务提供背景。

### 1.4 层级处理顺序

当上下文内容过多时，请按此顺序处理：

1. **粗读外部状态** → 我在大局中处于什么位置？
2. **阅读你的内存** → 我已经知道了什么？
3. **专注于任务上下文** → 我现在要做什么？
4. **综合分析** → 我之前的知识如何辅助完成当前任务？

### 1.5 执行作用域 (Execution Scope - 块调用)

如果你在 block 调用中运行，你会收到执行作用域信息：

```
Execution scope:
  execution_id: 43
  block: process
  depth: 3
  parent_execution_id: 42
```

**这些信息告诉你：**

| 字段 | 含义 |
|-------|---------|
| `execution_id` | 此特定块调用的唯一 ID |
| `block` | 你正在其中执行的 block 名称 |
| `depth` | 调用栈深度（1 表示第一层） |
| `parent_execution_id` | 调用方框架的 ID（用于作用域链） |

**如何使用：**

1. **包含在你的绑定输出中**：写入绑定 (binding) 时，在文件名和前置元数据 (frontmatter) 中包含 `execution_id`，以便虚拟机正确跟踪作用域。

2. **理解变量隔离**：你的绑定不会与同一 block 的其他调用发生冲突。如果 block 递归调用自身，每次调用都有自己的 `execution_id`。

3. **上下文引用已预解析**：虚拟机在向你传递上下文之前已解析了变量引用。你无需手动遍历作用域链——虚拟机已经帮你做好了。

**示例：** 如果一个递归的 `process` 块深度为 5，则存在 5 个独立的 `execution_id` 值，每个都有自己的局部绑定。你的会话仅能看到当前框架的上下文。

## 2. 处理持久化状态 (Working with Persistent State)

如果你是一个持久化代理，你需要通过内存文件跨会话维护状态。

### 两种截然不同的输出

持久化代理有**两种不同的输出**，绝不能混淆：

| 输出 | 是什么 | 去向 | 用途 |
|--------|------------|---------------|---------|
| **绑定 (Binding)** | “本次”任务的结果 | `bindings/{name}.md` 或数据库 | 通过 `context:` 传递给其他会话 |
| **内存 (Memory)** | 你积累的知识 | `agents/{name}/memory.md` 或数据库 | 传递给你“未来”的调用 |

**绑定是针对任务的。** 如果你被要求“审查计划”，绑定内容就是你的审查意见。

**内存是针对代理的。** 它包含你在所有调用中（而不只是这一次）积累的理解、决策和担忧。

它们被写入**不同的位置**，服务于**不同的目的**。请务必同时更新两者。

### 2.1 阅读你的内存 (Reading Your Memory)

在会话开始时，系统会提供你的内存文件。它包含：

- **当前理解 (Current Understanding)**：你对项目/任务的总体把握
- **已做决策 (Decisions Made)**：你决定了什么以及原因
- **待观察事项 (Open Concerns)**：你正在关注的问题
- **近期片段 (Recent Segments)**：最近几次会话发生了什么

**请仔细阅读。** 内存是你的连续性。一个忽略内存的持久化代理只是一个增加了额外步骤的无状态代理。

### 2.2 在先前知识基础上构建

当你遇到与内存相关的内容时：

- 明确引用它：“在我之前的审查中，我注意到 X……”
- 在其基础上构建：“鉴于我已经批准了计划，我现在正在检查实现的一致性……”
- 修正错误：“我之前认为 X，但现在我发现 Y……”

### 2.3 保持一致性

除非你明确改变立场，否则你的决策在不同片段之间应保持一致。如果你在第 1 片段批准了一个方案，不要在第 3 片段未经说明原因就否定同样的做法。

## 3. 内存压缩指南 (Memory Compaction Guidelines)

在会话结束时，你会被要求更新内存文件。这就是**压缩 (compaction)**——为未来的会话保留关键信息。

### 3.1 压缩不是简单的概括 (Summarization)

**错误做法：** “我审查了代码并发现了一些问题。”

这丢失了所有有用的信息。概括是泛化；压缩是保留细节。

**正确做法：** “审查了认证模块 (src/auth/login.ts:45-120)。发现：(1) 第 67 行查询构建器存在 SQL 注入风险，(2) 登录端点缺失频率限制，(3) 错误处理模式良好，值得复用。已要求修复第 1、2 点，总体结构已批准。”

### 3.2 保留什么

保留“未来的你”需要的**具体细节**：

| 保留内容 | 示例 |
| ---------------------------- | -------------------------------------------------------- |
| **具体位置** | 使用 "src/auth/login.ts:67" 而非 "认证代码" |
| **确切发现** | "查询构建器中的 SQL 注入" 而非 "安全问题" |
| **决策及其依据** | "因为 X 而批准" 而非仅仅 "批准" |
| **数字和阈值** | "覆盖率为 73%，目标是 80%" 而非 "覆盖率较低" |
| **名称和标识符** | "User.authenticate() 方法" 而非 "登录函数" |
| **待解决问题** | "待验证：频率限制是否适用于 OAuth 流程？" |

### 3.3 丢弃什么

丢弃对未来会话没有帮助的信息：

| 丢弃内容 | 原因 |
| ---------------- | --------------------------------------------------------------------------- |
| 推理链条 | 结论更重要，而非你如何得出结论的过程 |
| 错误的尝试 | 你考虑过 X 但选择了 Y——只需记录 Y 以及为什么不选 X 的简短说明 |
| 显而易见的上下文 | 不要重复任务提示词 (task prompt) 的内容 |
| 冗长的引用 | 通过位置引用，不要复制大段代码 |

### 3.4 压缩结构

请按此结构更新你的内存文件：

```markdown
## Current Understanding (当前理解)

[你对整个项目/任务的了解——更新，而非完全替换]

## Decisions Made (已做决策)

[追加带有日期和依据的新决策]

- [日期]: [决策] — [原因]

## Open Concerns (待观察事项)

[未来会话中需要留意的事项——增加新的，移除已解决的]

## Segment [N] Summary (第 N 片段摘要)

[“本次”会话发生了什么——要具体，不要泛泛而谈]

- 审查了：[什么，位置]
- 发现：[具体的发现]
- 决策：[具体的决策]
- 下一步：[接下来应该发生什么]
```

### 3.5 压缩示例

**糟糕的压缩（太笼统）：**

```
## Segment 3 Summary
审查了实现。发现了一些问题。要求修改。
```

**良好的压缩（具体且有用）：**

```
## Segment 3 Summary
- 审查了：步骤 2 的实现 (UserService.ts, AuthController.ts)
- 发现：
  - UserService.getById (第 34 行) 缺失空值检查
  - AuthController.login 未使用第 1 片段批准的错误格式
  - 优点：事务处理遵循了我推荐的模式
- 决策：在继续之前，要求修复空值检查和错误格式问题
- 下一步：修复后重新审查，然后批准进入步骤 3
```

### 3.6 具体性测试

在完成压缩之前，请自问：“如果我一周后只读这段摘要，我能确切理解发生了什么并做出连贯的后续决策吗？”

如果答案是否定的，请添加更多细节。

## 4. 上下文大小管理 (Context Size Management)

### 4.1 当内存变得太长时

经过多次片段，你的内存文件会增长。当它变得臃肿时：

1. **完整保留近期片段**（最后 2-3 个）
2. **压缩旧片段**，仅保留关键决策
3. **归档远古历史**，改为要点列表

```markdown
## Recent Segments (近期片段 - 完整细节)

[片段 7-9]

## Earlier Segments (早期片段 - 已压缩)

- 片段 4-6：完成初始实现审查，经小幅修正后批准
- 片段 1-3：建立审查标准，批准设计文档

## Key Historical Decisions (关键历史决策)

- 选择 JWT 而非 session tokens (片段 2)
- 建立 80% 代码覆盖率阈值 (片段 1)
```

### 4.2 当任务上下文很大时

如果你收到了巨大的任务上下文（大段代码、长文档）：

1. **不要试图全部记住** — 通过位置引用
2. **记录你检查了什么** — “审查了第 1-200 行，重点关注认证流程”
3. **记录具体位置** — 后续会话如有需要可以重新检查

## 5. 向虚拟机发出信号 (Signaling to the VM)

OpenProse 虚拟机通过阅读你的输出来确定后续步骤。请通过清晰的表达来协助它：

### 5.1 决策信号

当你做出影响控制流的决策时，请明确表述：

```
DECISION: 继续执行实现
RATIONALE: 计划解决了之前审查中提出的所有担忧
```

或

```
DECISION: 要求修订
ISSUES:
1. [具体问题]
2. [具体问题]
REQUIRED CHANGES: [需要进行哪些更改]
```

### 5.2 担忧信号

当你注意到不阻碍进度但需要跟踪的事项时：

```
CONCERN: [具体担忧]
SEVERITY: [低/中/高]
TRACKING: [需要观察什么]
```

### 5.3 完成信号

当你的片段完成时：

```
SEGMENT COMPLETE
MEMORY UPDATES:
- [需要添加到“当前理解”的内容]
- [需要记录的决策]
- [需要跟踪的担忧]
READY FOR: [接下来应该发生什么]
```

## 6. 编写输出文件 (Writing Output Files)

当使用基于文件的状态模式（参见 `../state/filesystem.md`）时，虚拟机会告诉你将输出写入何处。你必须将结果直接写入文件系统。

### 6.1 绑定输出文件 (Binding Output Files)

对于带有输出捕获的常规会话（`let x = session "..."`），请写入指定的绑定路径：

**路径格式：** `.prose/runs/{run-id}/bindings/{name}.md`

**路径格式（在块调用内部）：** `.prose/runs/{run-id}/bindings/{name}__{execution_id}.md`

**文件格式：**

````markdown
# {name}

kind: {let|const|output|input}
execution_id: {id}  # 如果在块调用中，请包含此项（根作用域则省略）

source:

```prose
{创建此绑定的源代码}
```
````

---

{你实际的输出内容在此}

````

**示例：**

```markdown
# research

kind: let

source:
```prose
let research = session: researcher
  prompt: "研究 AI 安全"
````

---

AI 安全研究涵盖几个关键领域：

1. **对齐 (Alignment)** - 确保 AI 系统追求预定目标
2. **健壮性 (Robustness)** - 使系统在边缘情况下保持韧性
3. **可解释性 (Interpretability)** - 理解模型如何做出决策

关键论文包括 Amodei 等 (2016) 关于具体问题的研究……

````

### 6.2 匿名会话输出 (Anonymous Session Output)

没有明确捕获的会话（只有 `session "..."` 而没有 `let x =`）依然会产生输出。这些以 `anon_` 前缀写入：

**路径：** `.prose/runs/{run-id}/bindings/anon_001.md`

虚拟机会分配连续的编号。请使用相同的格式，但注明绑定来自匿名会话：

```markdown
# anon_003

kind: let

source:
```prose
session "分析代码库中的安全问题"
````

---

安全分析发现了以下问题……

````

### 6.3 持久化代理内存输出 (Persistent Agent Memory Output)

如果你是一个持久化代理（通过 `resume:` 调用），你还有额外的责任：

1. **首先阅读你的内存文件**
2. **结合内存 + 上下文处理任务**
3. **使用压缩后的状态更新你的内存文件**
4. **编写一个片段 (segment) 文件**来记录本次会话

**内存文件路径：** `.prose/runs/{run-id}/agents/{name}/memory.md`（或用于项目范围的 `.prose/agents/{name}/`，或用于用户范围的 `~/.prose/agents/{name}/`）

**片段文件路径：** `.prose/runs/{run-id}/agents/{name}/{name}-{NNN}.md`

**内存文件格式：**

```markdown
# Agent Memory: {name} (代理内存)

## Current Understanding (当前理解)

{你对项目/任务积累的知识}

## Decisions Made (已做决策)

- {日期}: {决策} — {依据}
- {日期}: {决策} — {依据}

## Open Concerns (待观察事项)

- {担忧 1}
- {担忧 2}
````

**片段文件格式：**

```markdown
# Segment {NNN} (第 N 片段)

timestamp: {ISO8601}
prompt: "{本次会话的提示词}"

## Summary (摘要)

- 审查了：{你检查了什么}
- 发现：{具体的发现}
- 决策：{具体的决策}
- 下一步：{接下来应该发生什么}
```

### 6.4 输出编写检查清单

在完成会话之前：

- [ ] 将输出写入指定的绑定路径
- [ ] 如果是持久化代理：更新 memory.md
- [ ] 如果是持久化代理：编写片段文件
- [ ] 使用指定的精确文件格式
- [ ] 包含源代码片段以便溯源

## 7. 返回虚拟机 (Returning to the VM)

当你的会话完成时，你向虚拟机返回一个**确认消息**——而不是你的全部输出。虚拟机跟踪的是指针，而非具体值。

### 7.1 返回什么

你的返回消息应包括：

```
Binding written: {name} (已写入绑定)
Location: {路径或数据库坐标}
Summary: {关于绑定内容的 1-2 句摘要}
```

**示例（文件系统状态，根作用域）：**
```
Binding written: research
Location: .prose/runs/20260116-143052-a7b3c9/bindings/research.md
Summary: 详尽的 AI 安全研究，涵盖对齐、健壮性和可解释性，包含 15 篇关键论文引用。
```

**示例（文件系统状态，在块调用内部）：**
```
Binding written: result
Location: .prose/runs/20260116-143052-a7b3c9/bindings/result__43.md
Execution ID: 43
Summary: 将数据块处理为 3 个子部分，以进行递归处理。
```

**示例（PostgreSQL 状态）：**
```
Binding written: research
Location: openprose.bindings WHERE name='research' AND run_id='20260116-143052-a7b3c9'
Summary: 详尽的 AI 安全研究，涵盖对齐、健壮性和可解释性，包含 15 篇关键论文引用。
```

**示例（PostgreSQL 状态，在块调用内部）：**
```
Binding written: result
Location: openprose.bindings WHERE name='result' AND run_id='20260116-143052-a7b3c9' AND execution_id=43
Execution ID: 43
Summary: 将数据块处理为 3 个子部分，以进行递归处理。
```

### 7.2 为什么是指针，而非具体值

虚拟机从不在其工作内存中保存完整的绑定值。这是有意为之：

1. **可扩展性**：绑定内容可以非常大（兆字节甚至千兆字节）
2. **RLM 模式**：实现了“环境即变量”，代理可以通过编程方式查询状态
3. **上下文效率**：无论中间数据多大，虚拟机的上下文始终保持精简
4. **并发访问**：多个代理可以同时读写不同的绑定

### 7.3 不要返回什么

不要在 Task 工具的响应中返回你的完整输出。虚拟机会忽略它。

**错误做法：**
```
这是我的研究结果：

AI 安全是一个研究如何创建有益且避免有害结果的 AI 系统的领域。该领域涵盖几个关键领域……
[此处省略 5000 字]
```

**正确做法：**
```
Binding written: research
Location: .prose/runs/20260116-143052-a7b3c9/bindings/research.md
Summary: 5200 字的 AI 安全综述，涵盖对齐、健壮性、可解释性和治理，包含 15 处引用。
```

### 7.4 对于持久化代理

如果你是一个持久化代理（通过 `resume:` 调用），还需确认你的内存更新：

```
Binding written: analysis
Location: .prose/runs/20260116-143052-a7b3c9/bindings/analysis.md
Summary: 风险评估，识别出 3 个严重关注点和 5 个中度关注点。

Memory updated: captain
Location: .prose/runs/20260116-143052-a7b3c9/agents/captain/memory.md
Segment: captain-003.md
```

## 摘要 (Summary)

作为 OpenProse 程序中的一个子代理：

1. **理解你的上下文层级** — 外部状态、内存、任务上下文
2. **按引用读取上下文** — 直接访问绑定文件/数据库，仅加载你需要的内容
3. **在内存基础上构建** — 你拥有连续性，请善用它
4. **压缩而非概括** — 保留细节，丢弃推理链条
5. **清晰发出信号** — 协助虚拟机理解你的决策
6. **测试你的压缩质量** — “未来的你”能确切理解发生了什么吗？
7. **直接编写输出** — 持久化到分配给你的绑定位置
8. **返回指针而非具体值** — 虚拟机跟踪的是位置，而非内容

内存是让你具有持久性的关键。虚拟机的效率取决于你正确写入输出并返回确认消息——而不是通过底座倾倒全部内容。
